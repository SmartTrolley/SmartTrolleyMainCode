General Rules

NO MAGIC NUMBERS - please make all of them variables, even if they are only temporary (add a comment to the variable saying it is temporary, makes it easy to refactor [perhaps use a keyword, then ctrl+f and delete all those variables when refactoring])

Add an explanation if the code is complicated. Try writing simple code, simple solutions often work better.

Any Code to be completed/corrected (even if left overnight) should have a:

//TODO [Explanation of what is meant to happen/needs to be coded]

Save it to add the comment to your tasks list, making it easier to get to the code and understand what you wanted to do.

Try making sure all names are meaningful, and describe the function of that thing (class, method, variable etc).

Variable and Method names should use lower camel case i.e. the first letter of the first word is lower case, but the first letter of following words is upper case e.g. likeThisHere

Class and interface names should be in camel case, but the first letter is also capitalised e.g. LikeThisHere

Constant names (e.g. static final var_name) should be all upper case i.e. PI

CTRL+SHIFT+Y changes to lower case

CTRL+SHIFT+X changes to upper case

Package names are all lower case.

Tags (@...) are used to automatically generate Javadoc. If you want to include an image in a doc comment, place the image file in a doc-files subdirectory of the source code directory. Give the image the same name as the class, with an integer suffix. For example, the second image         that appears in the doc comment for a class named Circle can be included with this HTML tag: <IMG src="doc-files/Circle-2.gif">

Put a blank line before a comment if it has a code line following, and a blank line after a comment that follows a code line, to make it clear which line the comment applies to.

Use autoformat feature in Eclipse - ctrl+shift+F

Use autocomplete in Eclipse to help with coding - ctrl+space

Variables

DO NOT USE GLOBAL VARIABLES.

THE FOLLOWING APPLIES TO ALL VARIABLES (LOCAL, CLASS LEVEL [fields in Java speak] ETC)

Variables either have self-explanatory names or comments explaining their function (If a good name is too long). There should be no use of random variable names, and ensure any temporary variables (for testing purposes etc) have “temp” in their name or comment (to facilitate their removal at a later stage). Comments for fields should be “/** TEXT */”.

No magic numbers are to be used anywhere in the code. Use variables instead.

Variable names should use lower camel case i.e. the first letter of the first word is lower case, but the first letter of following words is upper case e.g. likeThisHere

Loops

All Loops must contain a comment at the start explaining the loop. It should at least give its purpose, the number of iterations and the variable iterated over.

Methods/Tests

Method names should be descriptive, can be fairly long, and should use lower camel case i.e. the first letter of the first word is lowercase, but the first letter of following words is upper case e.g. likeThisHere

All Methods must have a comment at the start detailing:

  1. Their (expected) function
  2. The test they satisfy
  3. The variables passed in, and those returned

Method Header:

/**

* Method/Test Description

* <p>Test(s)/User Story that it satisfies

*@param [parameter_type parameter_name parameter_description

*@param [List all input parameters]

* [If applicable]@exception [exception thrown] [exception description]

*@exception [Repeat for all exceptions thrown]

* @return [return_type] [return_var_name] [variable_description]

* [If applicable]@see [Reference URL OR Class#Method]

*<p> Date Modified: ${date}

*/

Classes

Every Class created should contain the following header (Tick “Generate Comments” when creating the class):

(Do not use double quotes or apostrophes in this header)

Class Header :

/**

* Workspace_Name[c][d]

*

* A DESCRIPTION OF THE CLASS

*

* @author [Name1]

* @author [Name2]

*

* @author [Checked By:] [Checker(s) fill here]

*

* @version [version of this class] [Date Created: DD/MM/YY]

*/

Package

Every Package should be created with a package-info (Tick “Create package-info.java” when creating a package) containing the following header.

(Do not use double quotes or apostrophes in this header)

Package-Info Header:

/**

* Workspace_Name

*

* A DESCRIPTION OF THE PACKAGE

*

* @author [Name1]

* @author [Name2]

*

* @author [Checked By:] [Checker(s) fill here]

*

* @version [version of this class] [DateCreated: DD/MM/YY]

*/

End of File

Add this after the end of all classes/packages.

/**************End of [Class Name]**************/

Default Package/Class Template for Eclipse:

By using this, the required comment block for packages/classes can be auto-generated by pressing ALT+SHIFT+J.

Comments generation is configured in:

Main Menu -> Window -> Preferences -> Java -> Code Style -> Code Templates -> Comments -> Files -> Edit

Check “Automatically add comments for new methods and types”

Then paste:

/**

* ${project_name}

*

* A DESCRIPTION OF THE FILE

*

* @author Name1

* @author Name2

*

* @author Checked By: Checker(s) fill here

*

* @version version of this file [Date Created: ${date}]

*/

/*YOUR CODE HERE*/

 

 

/**************End of ${file_name}**************/

Default Method Template for Eclipse

By using this, the required comment block for methods can be auto-generated by pressing ALT+SHIFT+J.

Comments generation is configured in:

Main Menu -> Window -> Preferences -> Java -> Code Style -> Code Templates -> Comments -> Methods-> Edit

Check “Automatically add comments for new methods and types”

Paste In:

/**

*Method/Test Description

*<p>Test(s)/User Story that it satisfies

*${tags}

*[If applicable]@see [Reference URL OR Class#Method]

*<p> Date Modified: ${date}

*/

REMEMBER THAT IF YOU CHANGE THE METHOD, PLEASE UPDATE THE COMMENT BLOCK. You can also delete the entire block, and regenerate it with alt+shift+j

Default Fields Template for Eclipse (Optional)

By using this, a comment block for fields can be auto-generated by pressing ALT+SHIFT+J.

Comments generation is configured in:

Main Menu -> Window -> Preferences -> Java -> Code Style -> Code Templates -> Comments -> Fields> Edit

Paste In:

/**Field Detail*/

Default Constructor Template for Eclipse (Optional)

By using this, a comment block for constructors can be auto-generated by pressing ALT+SHIFT+J.

Comments generation is configured in:

Main Menu -> Window -> Preferences -> Java -> Code Style -> Code Templates -> Comments -> Constructor> Edit

Paste In:

/**

 * DESCRIPTION OF CONSTRUCTOR

*<p> Date Modified: ${date}

 */

Keyboard Shortcuts in Eclipse

  1. CTRL+SHIFT+Y - changes to lower case
  2. CTRL+SHIFT+X - changes to upper case
  3. ALT+SHIFT+J - Generates Comment Block
  4. CTRL+SHIFT+O (When project is selected) - Removes unused imports and organises the project

Steps for Generating Javadoc

  1. File -> Export -> Java -> Javadoc [-> Next]
  2. Paste “C:\Program Files (x86)\Java\jdk1.7.0_51\bin\javadoc.exe” (OR your path to Javadoc.exe [should be in the same folder as other Java stuff]) into “Javadoc command” (If this does not work, search for “javadoc.exe” in your “C:\Program Files (x86)\Java” directory
  3. Select the packages you want to generate Javadoc for in “Select types for which Javadoc will be generated”
  4. In “Create Javadoc for members with visibility” check “Private”[e]
  5. Use the doclet fields to specify where you want to save the Javadoc (Default Recommended)  [-> Next]
  6. Make sure the document has a title, and everything in “Basic Options” is checked [-> Finish][f]
  7. Take a look in the console window for any errors
